dirg

A minimal grid library

The gist

  1. Download dirg (and possibly breakpoint).
  2. Put dirg somewhere in your build where whatever uses it has access to the functions and mixin it makes available.
  3. There is no step 3.

Scales whirlwind tour

A scale has a base font size value, a unit size that’s always equal or larger than it, and a variance factor.

The cente of the scale is 0. You can move to 1 or greater or -1 or lesser on the scale. The smaller the variance factor of the scale, the more fine grained variations you’ll get.

The default scale in dirg is looks like this: 

              $dirg-scales: (
                default: (
                  font-size: 14px,
                  unit: 14px * 1.5,
                  scale: 1.2
                )
              );

You can tweak that, or add further entries for smaller/larger screens, for instance. You should never mix two scales,however. 

            $dirg-scales: (
              default: (
                font-size: 14px,
                unit: 14px * 1.5,
                scale: 1.2
              ),
              small: (
                font-size: 12px,
                unit: 12px * 1.35,
                scale: 1.25
              )
            );

This is where the breakpoint mixin becomes useful. When you apply a different scale to sizing, it’s always because a change of resolution occurred (since different size screens have different sizing needs): 

          
            .a-component {
              .paragraph {
                @include font-size(1);

                @include breakpoint(small) {
                  // Scale value for 1, but on the small scale.
                  @include font-size(1, small);
                }
              }
            }

API

font-scale($x, [$scale = default]): Returns a value to be used on a `font-size`. 

            
              .a-component {
                // …

                .heading {
                  font-size: font-scale(3);
                }

                .paragraph {
                  font-size: font-scale(1);
                }
              }

units($x, [$scale = default]): Returns the value for xunits of the grid. Use this for sizing things: 

            
              .a-component {
                width: units(4);
              }

columns($x, [$scale = default]): Returns a column, which is the equivalent of units(4). This is really just a shortcut, but it tends to make discussions sane by avoiding too high values passed to units() that are hard to reason about what they mean. 

            
              .main-wrapper {
                width: columns(8);
              }

              // Elsewhere…
              .main-wrapper .login-pane {
                width: columns(2);
              }

              // And elsewhere
              .main-wrapper .content-wrapper {
                width: columns(6);
              }

@include font-size($x, [$scale = default]): A mixin for making setting font sizes with a grid-valid line height value that can accommodate it. 

            
              .a-component {
                .paragraph {
                  @include font-size(1);
                }
              }

Most elements/components are sized vertically by the amount of text and/or other elements in them. This ensures they will grow in grid steps, and thus, correctly.

The long story

Designs that don’t use a grid are fundamentally broken. Using a grid, however, in the majority of cases is painful because silly bears insist on making whole frameworks out of it. Learning the grid system itself becomes a science. That’s nuts.

Rhythm and proportion are fundamental principles of UI building. These need to be orderly and based on a scale, which determines how things like text can grow or shrink.

One side benefit to having a scale is, because the factor is predictable (i.e.: you know text going from 0 on to 1 on the scale needs to grow by x%), you can export this factor to CSS transforms and animate proportionally.

For instance, it’s a known fact you can’t performantly animate font-size. But you can use a scale() transform just fine. The problem then becomes knowing by exactly how much you can grow/shrink things with that transform so that it moves to a direction within the scale.
